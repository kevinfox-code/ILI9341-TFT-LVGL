# TouchGFX adapter notes

TouchGFX projects are generated by TouchGFX Designer, so this directory is
documentation and skeleton code, not a compiled library target — unlike
`adapters/lvgl/`, there is nothing here for `tft_drivers`'s CMakeLists to
build.

`drv_constants.h` and `DRV_Setup()` are identical to the LVGL case (see the
top-level README's "bring-up recipe"); only Layer 2 differs. Everything below
plugs into the `TouchGFXHAL`/`TouchGFXGeneratedHAL` and `TouchController`
classes that TouchGFX Designer generates into your project's
`TouchGFX/generated/` and `TouchGFX/target/` folders.

## Display flush: `HAL::flushFrameBuffer`

TouchGFX calls `HAL::flushFrameBuffer(const Rect& rect)` once per frame and
expects `HAL::endFrame()` semantics: kick off the transfer, then let
TouchGFX know when the panel has actually been updated via
`HAL::frameBufferReady()` (called from your flush-complete notification, not
synchronously). This lines up directly with `ILI9341_FlushAsync` +
`ILI9341_done_cb_t`.

```cpp
// TouchGFX/target/STM32TouchGFXHAL.hpp (or wherever your generated HAL lives)
class STM32TouchGFXHAL : public TouchGFXGeneratedHAL
{
public:
    STM32TouchGFXHAL(...) : TouchGFXGeneratedHAL(...), disp(nullptr) {}

    void setDisplay(ili9341_t* d) { disp = d; }

    virtual void flushFrameBuffer(const touchgfx::Rect& rect) override
    {
        TouchGFXGeneratedHAL::flushFrameBuffer(rect); // bookkeeping (dirty regions etc.)

        if (disp == nullptr) return;

        const uint16_t x0 = rect.x, y0 = rect.y;
        const uint16_t x1 = rect.x + rect.width - 1;
        const uint16_t y1 = rect.y + rect.height - 1;
        uint8_t* px = getTFTFrameBuffer() /* or your source buffer ptr */;
        uint32_t len = (uint32_t)rect.width * rect.height * 2u;

        drv_status_t st = ILI9341_FlushAsync(disp, x0, y0, x1, y1, px, len,
                                              &STM32TouchGFXHAL::onFlushDone, this);
        if (st != DRV_OK) {
            // Fall back to a blocking flush rather than stalling the frame.
            ILI9341_Flush(disp, x0, y0, x1, y1, px, len, 100);
            frameBufferReady();
        }
    }

private:
    static void onFlushDone(void* user)
    {
        // ISR context — do the minimum, defer real work to the TouchGFX task
        // the same way the generated HAL's DMA2D/LTDC completion path does.
        static_cast<STM32TouchGFXHAL*>(user)->frameBufferReady();
    }

    ili9341_t* disp;
};
```

If you need a blocking flush at a given call site (e.g. during startup before
interrupts are configured), call `ILI9341_Flush(...)` directly and then
`frameBufferReady()` synchronously — never call both for the same rect.

`HAL::endFrame()` itself needs no changes: TouchGFX calls it after
`flushFrameBuffer` returns, and `frameBufferReady()` (invoked from
`onFlushDone`, above) is what tells TouchGFX's task the panel write
completed, exactly like the generated DMA2D path it replaces.

## Touch input: `TouchController::sampleTouch`

```cpp
// TouchGFX/target/generated/XPT2046TouchController.hpp
class XPT2046TouchController : public touchgfx::TouchController
{
public:
    void setTouch(xpt2046_t* t, ili9341_t* d) { touch = t; disp = d; }

    virtual void init() override { /* DRV_Setup() already initialized the chip */ }

    virtual bool sampleTouch(int32_t& x, int32_t& y) override
    {
        if (touch == nullptr || disp == nullptr) return false;

        uint16_t hor_res, ver_res;
        ILI9341_GetResolution(disp, &hor_res, &ver_res);

        uint16_t px, py;
        // rotation: whatever DRV_DISP_ROTATION resolved to for this build.
        if (!XPT2046_ReadPoint(touch, &px, &py, hor_res, ver_res, DRV_DISP_ROTATION)) {
            return false;
        }
        x = px;
        y = py;
        return true;
    }

private:
    xpt2046_t* touch = nullptr;
    ili9341_t* disp = nullptr;
};
```

`sampleTouch` is polled from the TouchGFX task (same threading model as
LVGL's `read_cb`), so no additional locking is needed beyond what
`XPT2046_ReadPoint`/`drv_spi_bus_lock` already provide. `DRV_DISP_ROTATION`
comes from `drv_constants.h`; reference it directly, or thread the value
through from `DRV_GetDisplay()`'s config if you support runtime rotation
changes.

## Shared bring-up

Both adapters call the exact same `DRV_Setup()` / `DRV_GetDisplay()` /
`DRV_GetTouch()` from `drv_setup.h`, and rely on the same `DRV_ISR_*` wiring
from Phase 3 (`drv_isr_glue.c` or your own HAL callback forwarding). Only the
two classes above — the HAL flush override and the `TouchController`
subclass — are TouchGFX-specific.
