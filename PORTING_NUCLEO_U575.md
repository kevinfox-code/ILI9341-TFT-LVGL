# Porting Guide — NUCLEO-U575ZI-Q as `Profiles/board_2`

**Goal:** bring a second board profile to parity with `Profiles/board_1`
(NUCLEO-F446RE) at the *driver* level — a working display + touch + FreeRTOS
build, host tests still green, layering check still passing, docs updated.

**Scope note:** `board_1`'s current app GUI is TouchGFX (vendored under
`Profiles/board_1/TouchGFX/`, generated by TouchGFX Designer;
`DRV_ADAPTER_LVGL` is `OFF` in `App/CMakeLists.txt`). Redoing that TouchGFX
Designer bring-up for the U575 is a separate, GUI-tool-driven task and is
**out of scope here** — this guide proves the U575 port using
`tft_drivers`'s LVGL adapter instead, since it's already MCU-portable and
needs no external tool. `App/gui/gui.c` + `App/lv_conf.h` are still in the
tree (unused by the current TouchGFX build) and are exactly what you'd
re-enable to prove `board_2` end-to-end; swap to TouchGFX for `board_2`
afterward, following `App/drivers/adapters/touchgfx/README.md`, once you
want full parity.

Reference [README.md](README.md) and
[App/drivers/README.md](App/drivers/README.md) for everything not
board-specific (the `drv_constants.h` contract, the `drv_platform` CMake
contract, interrupt wiring, rotation reference). This document only calls
out what differs for the U575, and the one piece of real CMake work needed
because this is a cross-family port (not just a new same-family board).

`tft_drivers` itself (`App/drivers/CMakeLists.txt`, `src/`, `include/drv/`)
is already MCU-agnostic — nothing in it names an STM32 family. Only two
files in the **consuming project** assume STM32F4xx and aren't
parameterized by `${PROFILE}` today, because there's only ever been one
profile:

1. **`Profiles/board_1/cmake/gcc-arm-none-eabi.cmake`** — hardcodes
   `-mcpu=cortex-m4 -mfpu=fpv4-sp-d16 -mfloat-abi=hard` and the linker
   script filename `STM32F446XX_FLASH.ld`. `${PROFILE}` only picks the
   *directory* here, not the MCU flags or script name.
2. **`App/CMakeLists.txt`** — the `drv_platform` interface target
   unconditionally points at `Drivers/CMSIS/Device/ST/STM32F4xx/Include`,
   `Drivers/STM32F4xx_HAL_Driver/Inc(+Legacy)`,
   `Middlewares/.../portable/GCC/ARM_CM4F`, and defines `STM32F446xx`.

Plan: give `board_2` its own toolchain file, and pull `App/CMakeLists.txt`'s
family-specific bits out into a per-profile CMake fragment. Everything else
below is CubeMX + a new `drv_constants.h`.

## 1. Decisions to make before CubeMX

- **Exact part number.** "Nucleo U575" is `NUCLEO-U575ZI-Q`, MCU
  `STM32U575ZITxQ`. Confirm on the board silkscreen — the compile define
  (`STM32U575xx`) and CMSIS device header must match exactly.
- **TrustZone: disable it.** The U575 is TrustZone-capable; for parity with
  board_1 (single non-secure image, no secure/non-secure split), explicitly
  set **Project Manager → Security → TrustZone: Disabled** (`TZEN=0`).
  Leaving TrustZone on materially complicates the FreeRTOS port variant,
  linker script, and boot flow for no benefit to this driver bring-up.
- **FreeRTOS port variant.** With TrustZone disabled, CubeMX generates the
  `GCC/ARM_CM33_NTZ/non_secure` FreeRTOS port — not `ARM_CM4F` (board_1) and
  not `ARM_CM33`/`ARM_CM33_TZ` (those are the TrustZone variants).

## 2. CubeMX peripheral setup

Same shape as `App/drivers/README.md`'s CubeMX section; only family-specific
values differ.

### Clock configuration
1. STM32U575 uses a different clock tree (MSIS/MSIK + PLL, VOS ranges) than
   the F4's HSE+PLL — use CubeMX's Clock Configuration tab to solve for max
   SYSCLK (160 MHz) rather than hand-copying the F446 PLL values.
2. Set **Power → Regulator Voltage Scaling → Range 1** to support the
   higher SYSCLK.
3. **Enable ICACHE.** Cortex-M33 has no internal flash prefetch buffer like
   the M4; without it, flash wait-state penalties are severe. Verify it's
   on (CubeMX usually defaults it on for U5).
4. Verify SysTick is still HCLK-derived (`HAL_Delay`/`HAL_GetTick`).

### SPI for the Display (ILI9341) and Touch (XPT2046)
Same electrical requirements as board_1 (display: Full-Duplex Master, 8-bit,
CPOL Low/CPHA 1 Edge, MSB First, DMA on TX; touch: same mode, no DMA,
≤2 MHz). Differences:
- NUCLEO-U575ZI-Q is a **Nucleo-144** board (Zio + Morpho), not Nucleo-64 —
  the SPI instance/pins you wire to CS/DC/RESET will differ from board_1's
  SPI1/SPI2. Pick any DMA-capable SPI instance for the display; label pins
  the same way (`CS`, `DC`, `RESET`, `T_CS`, `T_IRQ`) so `drv_constants.h`
  reads consistently across profiles.
- STM32U5 uses **GPDMA1**, not the F4's DMA1/DMA2 stream model — add the
  SPI_TX request under GPDMA1 channel config. The HAL call sequence
  (`HAL_SPI_Transmit_DMA`, `TxCpltCallback`) is unchanged; `drv_spi_stm32.c`
  only calls the SPI HAL API, so no driver-side change is needed.
- NVIC priority for the SPI/DMA interrupt and the `T_IRQ` EXTI line still
  must be numerically ≥ `configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY` —
  re-derive the actual number for M33 (see FreeRTOS step below), don't
  assume board_1's `5` carries over.
- Recommended: enable "Register Callback" for the display SPI, same as
  board_1, so `DRV_Setup()` self-registers instead of relying on the weak
  `HAL_SPI_TxCpltCallback` in `App/drivers/src/drv_isr_glue.c`.

### GPIO pins
Same signal set as board_1 (`CS`/`DC`/`RESET`/`T_CS` as GPIO_Output
push-pull no-pull high-speed; `T_IRQ` as GPIO_EXTI falling-edge). Locations
are whatever's free on the Zio/Morpho headers you wire the panel to.

### FreeRTOS
1. **Middleware → FreeRTOS**, CMSIS-RTOS V2 API — same as board_1.
2. Heap_4, ≥32 KB, `USE_MUTEXES`/`USE_RECURSIVE_MUTEXES`/
   `USE_COUNTING_SEMAPHORES` all on — same as board_1.
3. Confirm CubeMX selects the `ARM_CM33_NTZ/non_secure` port once TrustZone
   is disabled (force it if the Middleware "Port" dropdown doesn't
   auto-select).
4. Don't add GUI/app tasks to CubeMX's FreeRTOS task list — same rule as
   board_1; they belong in `App_CreateTasks()`.
5. In `Core/Inc/FreeRTOSConfig.h`, verify:
   ```c
   #define configENABLE_FPU                              1     /* M33 has an FPU on U575 */
   #define configENABLE_MPU                               0
   #define configENABLE_TRUSTZONE                         0     /* TZEN=0 */
   #define configTOTAL_HEAP_SIZE                         (32 * 1024)
   #define configTICK_RATE_HZ                             1000
   #define configUSE_MUTEXES                               1
   #define configUSE_RECURSIVE_MUTEXES                     1
   #define configUSE_COUNTING_SEMAPHORES                   1
   #define configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY    <re-derive for M33 — priority-bit grouping differs from M4, don't hardcode board_1's 5>
   ```

### Optional: RTC, UART, IWDG, CRC
`board_1`'s `App/drv_constants.h` has an `APP_*` section
(`APP_RTC_HANDLE`, `APP_UART_HANDLE`, `APP_DEBUG_LED_PORT/_PIN`,
`APP_IWDG_HANDLE`) consumed only by `App/system.c`, for the demo clock/UART
debug/watchdog features. Only configure these in CubeMX if you want that
same demo functionality on `board_2`:
- **RTC**: LSE (accurate) or LSI (bring-up without a crystal).
- **UART with DMA**: check NUCLEO-U575ZI-Q's User Manual (UM2861) for which
  USARTx is wired to the ST-LINK Virtual COM Port — it's not necessarily
  USART2 like board_1.
- **IWDG**: same peripheral concept, no family-specific gotchas.
- **CRC**: enable the CRC peripheral if `App/system.c`'s CRC usage (check
  current `App/system.c` for what it's used for) is something you want to
  carry over.

### Generate code
**Toolchain/IDE = CMake**, target location `Profiles/board_2/`, **Generate
Code**.

## 3. Repository changes

### 3.1 New profile directory
CubeMX generates directly into `Profiles/board_2/` — no manual copy needed.
Sanity-check the generated tree matches board_1's shape (`Core/{Inc,Src}`,
`Drivers/CMSIS`, `Drivers/STM32U5xx_HAL_Driver`,
`Middlewares/Third_Party/FreeRTOS`, `cmake/stm32cubemx/CMakeLists.txt`, a
`.ioc`, a linker script, a startup `.s` file). Copy
`Profiles/board_1/Core/Src/cmsis_os2_ext.c` into
`Profiles/board_2/Core/Src/cmsis_os2_ext.c` unchanged — it's a hand-added
CMSIS-RTOS2 shim (`osThreadDetach` stub), not CubeMX output, and the
top-level `CMakeLists.txt` compiles it unconditionally for whichever
`${PROFILE}` is selected.

### 3.2 `App/drv_constants.h` — make it a per-profile switch
Today there's one `App/drv_constants.h` for board_1 only. Turn it into a
one-line switch on the MCU define (which `drv_platform`, see §3.3, already
provides):

```c
#ifndef DRV_CONSTANTS_H_
#define DRV_CONSTANTS_H_
#if defined(STM32F446xx)
#include "drv_constants_board_1.h"
#elif defined(STM32U575xx)
#include "drv_constants_board_2.h"
#else
#error "No drv_constants_*.h mapped for this MCU define"
#endif
#endif
```

Move the existing file's content verbatim into
`App/drv_constants_board_1.h`, then write `App/drv_constants_board_2.h` for
the U575 — same macro set, new values (HAL header, init hooks, handles,
pins, EXTI line, resolution/rotation per your panel wiring; leave the touch
calibration mins/maxes at template defaults until you run
`lv_xpt2046_calibrate` on real hardware):

```c
#define DRV_HAL_HEADER               "stm32u5xx_hal.h"
#define DRV_DISP_SPI_MX_INIT()       MX_SPIx_Init()   /* whatever CubeMX named your display SPI */
#define DRV_TOUCH_SPI_MX_INIT()      MX_SPIy_Init()
#define DRV_DISP_SPI_HANDLE          (&hspix)
/* ... CS/DC/RESET, T_CS/T_IRQ, EXTI line from your GPIO step ... */
#define DRV_DISP_HOR_RES             320u   /* or 240u — see rotation reference in App/drivers/README.md */
#define DRV_DISP_VER_RES             240u
#define DRV_DISP_ROTATION            3u
```
If you configured the optional RTC/UART/IWDG peripherals, add the matching
`APP_*` macros too (same names as board_1's, new handles).

### 3.3 `App/CMakeLists.txt` — parameterize the family bits
Extract board_1's hardcoded family bits into
**`Profiles/board_1/cmake/platform.cmake`** (new):
```cmake
target_include_directories(drv_platform INTERFACE
    ${PROFILE_ROOT}/Drivers/CMSIS/Device/ST/STM32F4xx/Include
    ${PROFILE_ROOT}/Drivers/STM32F4xx_HAL_Driver/Inc
    ${PROFILE_ROOT}/Drivers/STM32F4xx_HAL_Driver/Inc/Legacy
    ${PROFILE_ROOT}/Middlewares/Third_Party/FreeRTOS/Source/portable/GCC/ARM_CM4F
)
target_compile_definitions(drv_platform INTERFACE STM32F446xx USE_HAL_DRIVER)
```

Add **`Profiles/board_2/cmake/platform.cmake`** (new — verify the exact
folder names against what CubeMX actually generated, U5 naming occasionally
differs from the `STM32<family>xx_HAL_Driver` pattern):
```cmake
target_include_directories(drv_platform INTERFACE
    ${PROFILE_ROOT}/Drivers/CMSIS/Device/ST/STM32U5xx/Include
    ${PROFILE_ROOT}/Drivers/STM32U5xx_HAL_Driver/Inc
    ${PROFILE_ROOT}/Middlewares/Third_Party/FreeRTOS/Source/portable/GCC/ARM_CM33_NTZ/non_secure
)
target_compile_definitions(drv_platform INTERFACE STM32U575xx USE_HAL_DRIVER)
```

In `App/CMakeLists.txt`, replace the hardcoded block:
```cmake
target_include_directories(drv_platform INTERFACE
    ${PROFILE_ROOT}/Drivers/CMSIS/Device/ST/STM32F4xx/Include
    ${PROFILE_ROOT}/Drivers/STM32F4xx_HAL_Driver/Inc
    ${PROFILE_ROOT}/Drivers/STM32F4xx_HAL_Driver/Inc/Legacy
    ${PROFILE_ROOT}/Middlewares/Third_Party/FreeRTOS/Source/portable/GCC/ARM_CM4F
)
target_compile_definitions(drv_platform INTERFACE STM32F446xx USE_HAL_DRIVER)
```
with:
```cmake
include(${PROFILE_ROOT}/cmake/platform.cmake)   # family-specific include dirs/defines + FreeRTOS port dir
```
(the include dirs that *are* common across families — `Core/Inc`,
`Drivers/CMSIS/Include`, `Middlewares/.../include`,
`Middlewares/.../CMSIS_RTOS_V2` — stay directly in `App/CMakeLists.txt`;
only what's genuinely family-specific moves into `platform.cmake`). Also
flip the two `set(...CACHE BOOL ... FORCE)` lines to prove the driver layer
via LVGL for this bring-up, per the scope note above:
```cmake
set(DRV_USE_CMSIS_RTOS2 ON CACHE BOOL "" FORCE)
set(DRV_ADAPTER_LVGL ON CACHE BOOL "" FORCE)   # was OFF (TouchGFX-only) on board_1
```
and add `gui/gui.c` back to the `app` library's sources, and
`add_subdirectory` LVGL (`App/lvgl/`) + `target_link_libraries(lvgl PUBLIC
drv_platform)` — this app lib config existed on this project's
`rework/layered-drivers` history before the TouchGFX pivot; resurrect it
from there if it's simpler than rewriting from scratch (`git show
rework/layered-drivers:examples/LVGL_Lesson/01_Display/App/CMakeLists.txt`
for reference — note that file's paths are from before the repo-root
flattening, so adapt paths, not structure).

### 3.4 Toolchain file — give board_2 its own
Copy `Profiles/board_1/cmake/gcc-arm-none-eabi.cmake` to
`Profiles/board_2/cmake/gcc-arm-none-eabi.cmake` and change:
```cmake
set(TARGET_FLAGS "-mcpu=cortex-m33 -mfpu=fpv5-sp-d16 -mfloat-abi=hard ")
...
set(CMAKE_EXE_LINKER_FLAGS "${CMAKE_EXE_LINKER_FLAGS} -T \"${CMAKE_SOURCE_DIR}/Profiles/${PROFILE}/<name-CubeMX-generated>.ld\"")
```
(the `.ld` filename is whatever CubeMX emits for the exact U575 part — check
`Profiles/board_2/` after generation). Everything else in the file (the
`find_program` toolchain discovery, linker-flag scaffolding,
`--specs=nano.specs`) is toolchain plumbing, not MCU-specific — leave as-is.

### 3.5 `CMakePresets.json` — add a board_2 configuration
Add a second hidden base preset alongside `default`, then Debug/Release
variants that inherit it, so `-DPROFILE=board_1` builds stay untouched:
```json
{
    "name": "board_2-default",
    "hidden": true,
    "generator": "Ninja",
    "binaryDir": "${sourceDir}/build/${presetName}",
    "toolchainFile": "${sourceDir}/Profiles/board_2/cmake/gcc-arm-none-eabi.cmake",
    "cacheVariables": { "PROFILE": "board_2" }
},
{
    "name": "board_2-Debug",
    "inherits": "board_2-default",
    "cacheVariables": { "CMAKE_BUILD_TYPE": "Debug" }
},
{
    "name": "board_2-Release",
    "inherits": "board_2-default",
    "cacheVariables": { "CMAKE_BUILD_TYPE": "Release" }
}
```
plus matching `buildPresets` entries. Leave the existing un-suffixed
`Debug`/`Release` presets pointed at board_1 (the default).

### 3.6 Wiring `App_Init()` / `App_CreateTasks()`
Identical mechanics to board_1 (see
`App/drivers/README.md#wiring-app_init--app_createtasks`):
- `Profiles/board_2/Core/Src/main.c`, inside `/* USER CODE BEGIN 2 */`
  (after peripheral init, before `osKernelStart()`): call `App_Init();`.
- `Profiles/board_2/Core/Src/freertos.c`'s `MX_FREERTOS_Init()`, inside
  `/* USER CODE BEGIN RTOS_THREADS */`: call `App_CreateTasks();`.
- One `#include "app_main.h"` in each file's own `USER CODE BEGIN Includes`
  block. `stm32u5xx_it.c` needs no edits.

## 4. Build

```bash
cmake --preset board_2-Debug
cmake --build --preset board_2-Debug
```

Expect a real iteration loop: missing-include errors point at whatever
`platform.cmake` paths don't match CubeMX's actual U5 folder names (§3.3),
and any leftover `-mcpu=cortex-m4`/wrong linker-script name (§3.4) shows up
as a compiler error (bad `-mcpu`) or a link error (missing `.ld` / MEMORY
region overflow at the wrong flash size).
